What confidence represents
confidence answers a different question than seat_probability:

seat_probability = "how likely is a seat free?" — a property of the world.
confidence = "how much should you trust that number?" — a property of the estimate itself (meta-level).
They're independent. Examples:

seat_probability = 0.85, confidence = high → "we're sure it's ~85%."
seat_probability = 0.85, confidence = low → "best guess 85%, but shaky."
So confidence is how much real information backs this bucket, driven by:

source / horizon — live (mostly observed facts) = high; blend (live + forecast) = medium; baseline (pure forecast, far-future) = low. (This is AD22: variance shrinks toward now.)
Band width — narrow p10–p90 = high; wide = low.
Degraded mode (NFR3) — if the live feed drops and a near-term bucket falls back to baseline, confidence drops to flag the degradation.
How it's utilized downstream (Scoring API / app layer)
Message framing — high → show a crisp number ("seat very likely"); low → hedge ("based on typical patterns, may vary").
Whether to surface at all — the app may suppress or heavily caveat a low-confidence prediction rather than mislead.
Graceful degradation signal (NFR3) — a sudden drop to low confidence on near-term buckets tells the app the live feed is down, so it can warn the user / fall back gracefully.
Ranking input (UJ1) — when scoring candidate flights, the app can prefer or annotate options whose occupancy estimate is high-confidence, rather than betting on a shaky far-future guess.
Monitoring/ops — persistently low confidence where it shouldn't be flags model/data issues (NFR4).
Crucially, the assembler emits confidence; how to act on it is the app layer's call (FR12) — same predict-and-score boundary we've kept throughout.
